\doxysubsection{Detailed Description}
\subsubsection*{template$<$typename T$>$\newline
class libcamera\+::\+Control$<$ T $>$}
Describe a control and its intrinsic properties. 

The \doxylink{classlibcamera_1_1Control}{Control} class models a control exposed by an object. Its template type name T refers to the control data type, and allows functions that operate on control values to be defined as template functions using the same type T for the control value. See for instance how the \doxylink{classlibcamera_1_1ControlList_a8333b8978844aec280086855af72a9c5}{Control\+List\+::get()} function returns a value corresponding to the type of the requested control.

While this class is the main means to refer to a control, the control identifying information is stored in the non-\/template base \doxylink{classlibcamera_1_1ControlId}{Control\+Id} class. This allows code that operates on a set of controls of different types to reference those controls through a \doxylink{classlibcamera_1_1ControlId}{Control\+Id} instead of a \doxylink{classlibcamera_1_1Control}{Control}. For instance, the list of controls supported by a camera is exposed as \doxylink{classlibcamera_1_1ControlId}{Control\+Id} instead of \doxylink{classlibcamera_1_1Control}{Control}.

Controls of any type can be defined through template specialisation, but libcamera only supports the bool, uint8\+\_\+t, int32\+\_\+t, int64\+\_\+t and float types natively (this includes types that are equivalent to the supported types, such as int and long int).

Controls IDs shall be unique. While nothing prevents multiple instances of the \doxylink{classlibcamera_1_1Control}{Control} class to be created with the same ID for the same object, doing so may cause undefined behaviour.

\doxysubsection{Detailed Description}
A frame capture request. 

A \doxylink{classlibcamera_1_1Request}{Request} allows an application to associate buffers and controls on a per-\/frame basis to be queued to the camera device for processing.

Add a \doxylink{classlibcamera_1_1FrameBuffer}{Frame\+Buffer} with its associated \doxylink{classlibcamera_1_1Stream}{Stream} to the \doxylink{classlibcamera_1_1Request}{Request}. 


\begin{DoxyParams}[1]{Parameters}
\mbox{\texttt{ in}}  & {\em stream} & The stream the buffer belongs to \\
\hline
\mbox{\texttt{ in}}  & {\em buffer} & The \doxylink{classlibcamera_1_1FrameBuffer}{Frame\+Buffer} to add to the request \\
\hline
\mbox{\texttt{ in}}  & {\em fence} & The optional fence\\
\hline
\end{DoxyParams}
A reference to the buffer is stored in the request. The caller is responsible for ensuring that the buffer will remain valid until the request complete callback is called.

A request can only contain one buffer per stream. If a buffer has already been added to the request for the same stream, this function returns -\/EEXIST.

A \doxylink{classlibcamera_1_1Fence}{Fence} can be optionally associated with the {\itshape buffer}.

When a valid \doxylink{classlibcamera_1_1Fence}{Fence} is provided to this function, {\itshape fence} is moved to {\itshape buffer} and this \doxylink{classlibcamera_1_1Request}{Request} will only be queued to the device once the fences of all its buffers have been correctly signalled.

If the {\itshape fence} associated with {\itshape buffer} isn\textquotesingle{}t signalled, the request will fail after a timeout. The buffer will still contain the fence, which applications must retrieve with \doxylink{classlibcamera_1_1FrameBuffer_a6312ef39ae7cf1d4f5c7b22d56e0e227}{Frame\+Buffer\+::release\+Fence()} before the buffer can be reused in another request. Attempting to add a buffer that still contains a fence to a request will result in this function returning -\/EEXIST.

Retrieve the request\textquotesingle{}s \doxylink{classlibcamera_1_1ControlList}{Control\+List}. 

Requests store a list of controls to be applied to all frames captured for the request. They are created with an empty list of controls that can be accessed through this function. \doxylink{classlibcamera_1_1Control}{Control} values can be retrieved using \doxylink{classlibcamera_1_1ControlList_a8333b8978844aec280086855af72a9c5}{Control\+List\+::get()} and updated using \doxylink{classlibcamera_1_1ControlList_aa41c425f3075349747fdbfc1e3a6fc46}{Control\+List\+::set()}.

Only controls supported by the camera to which this request will be submitted shall be included in the controls list. Attempting to add an unsupported control causes undefined behaviour.

Retrieve the sequence number for the request. 

When requests are queued, they are given a sequential number to track the order in which requests are queued to a camera. This number counts all requests given to a camera and is reset to zero between camera stop/start sequences.

It can be used to support debugging and identifying the flow of requests through a pipeline, but does not guarantee to represent the sequence number of any images in the stream. The sequence number is stored as an unsigned integer and will wrap when overflowed.

Retrieve the request completion status. 

The request status indicates whether the request has completed successfully or with an error. When requests are created and before they complete the request status is set to Request\+Pending, and is updated at completion time to Request\+Complete. If a request is cancelled at capture stop before it has completed, its status is set to Request\+Cancelled.

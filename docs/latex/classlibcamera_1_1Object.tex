\doxysubsection{Detailed Description}
Base object to support automatic signal disconnection. 

The \doxylink{classlibcamera_1_1Object}{Object} class simplifies signal/slot handling for classes implementing slots. By inheriting from \doxylink{classlibcamera_1_1Object}{Object}, an object is automatically disconnected from all connected signals when it gets destroyed.

\doxylink{classlibcamera_1_1Object}{Object} instances are bound to the thread of their parent, or the thread in which they\textquotesingle{}re created when they have no parent. When a message is posted to an object, its handler will run in the object\textquotesingle{}s thread. This allows implementing easy message passing between threads by inheriting from the \doxylink{classlibcamera_1_1Object}{Object} class.

Deleting an object from a thread other than the one the object is bound to is unsafe, unless the caller ensures that the object\textquotesingle{}s thread is stopped and no parent or child of the object gets deleted concurrently. See \doxylink{classlibcamera_1_1Object_ad52c17ff4161b166ae71e766d0649b97}{Object\+::\texorpdfstring{$\sim$}{\string~}\+Object()} for more information.

\doxylink{classlibcamera_1_1Object}{Object} slots connected to signals will also run in the context of the object\textquotesingle{}s thread, regardless of whether the signal is emitted in the same or in another thread.

Objects can be connected to multiple signals, but they can only be connected to each signal once. Attempting to create multiple concurrent connections between the same signal and the same \doxylink{classlibcamera_1_1Object}{Object} (to either the same or differents slots of the object) will cause an assertion failure. While it would be possible to allow the implementation to let objects connect to the same signal multiple times, there are no expected use cases for this in libcamera and this behaviour is restricted to favour defensive programming.

Deleting an \doxylink{classlibcamera_1_1Object}{Object} automatically disconnects all signals from the \doxylink{classlibcamera_1_1Object}{Object}\textquotesingle{}s slots. All the \doxylink{classlibcamera_1_1Object}{Object}\textquotesingle{}s children are made orphan, but stay bound to their current thread.

\doxylink{classlibcamera_1_1Object}{Object} instances shall be destroyed from the thread they are bound to, otherwise undefined behaviour may occur. If deletion of an \doxylink{classlibcamera_1_1Object}{Object} needs to be scheduled from a different thread, \doxylink{classlibcamera_1_1Object_a7432a52a2e69a29cfe8144929686d1e6}{delete\+Later()} shall be used.

As an exception to this rule, \doxylink{classlibcamera_1_1Object}{Object} instances may be deleted from a different thread if the thread the instance is bound to is stopped through the whole duration of the object\textquotesingle{}s destruction, {\itshape and} the parent and children of the object do not get deleted concurrently. The caller is responsible for fulfilling those requirements.

In all cases \doxylink{classlibcamera_1_1Object}{Object} instances shall be deleted before the \doxylink{classlibcamera_1_1Thread}{Thread} they are bound to.

{\footnotesize\ttfamily void libcamera\+::\+Object\+::delete\+Later (\begin{DoxyParamCaption}{ }\end{DoxyParamCaption})}



Schedule deletion of the instance in the thread it belongs to. 

This function schedules deletion of the \doxylink{classlibcamera_1_1Object}{Object} when control returns to the event loop that the object belongs to. This ensures the object is destroyed from the right context, as required by the libcamera threading model.

If this function is called before the thread\textquotesingle{}s event loop is started or after it has stopped, the object will be deleted when the event loop (re)starts. If this never occurs, the object will be leaked.

Deferred deletion can be used to control the destruction context with shared pointers. An object managed with shared pointers is deleted when the last reference is destroyed, which makes difficult to ensure through software design which context the deletion will take place in. With a custom deleter for the shared pointer using \doxylink{classlibcamera_1_1Object_a7432a52a2e69a29cfe8144929686d1e6}{delete\+Later()}, the deletion can be guaranteed to happen in the thread the object is bound to.

This function invokes the member method {\itshape func} with arguments {\itshape args}, based on the connection {\itshape type}. Depending on the type, the method will be called synchronously in the same thread or asynchronously in the object\textquotesingle{}s thread.

Arguments {\itshape args} passed by value or reference are copied, while pointers are passed untouched. The caller shall ensure that any pointer argument remains valid until the method is invoked.

Due to the asynchronous nature of threads, functions invoked asynchronously with the Connection\+Type\+Queued type are not guaranteed to be called before the thread is stopped. See \doxysectlink{classlibcamera_1_1Thread_thread-stop}{Stopping Threads}{1} for additional information.

This function posts the message {\itshape msg} to the message queue of the object\textquotesingle{}s thread, to be delivered to the object through the \doxylink{classlibcamera_1_1Object_a5328cab1412743716aa2748eeded8e80}{message()} function in the context of its thread. \doxylink{classlibcamera_1_1Message}{Message} ownership is passed to the thread, and the message will be deleted after being delivered.
